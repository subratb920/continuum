# Continuum Manifesto v1.0

---

## 1. Purpose

Continuum defines a disciplined execution model for a single individual operating under constrained time, energy, and availability.

The system is designed to:

- minimize re-entry friction
- externalize memory
- prevent scope expansion during execution
- enforce continuity over intensity

Continuum prioritizes returnability and consistency over speed and volume.

---

## 2. Core Principles

- Execution must be resumable without relying on memory.
- Planning must be isolated from execution.
- Work must progress through small, bounded units.
- Stopping must be safe and deterministic.
- The system must function under partial availability.

---

## 3. Execution Unit

The fundamental execution unit is an **Interval**.

An Interval:

- lasts 30–45 minutes
- operates on exactly one story
- has exactly one Session Goal
- produces a visible artifact
- ends with a Bridge

If work cannot be completed within an Interval, it must be decomposed.

---

## 4. Session Goal

Each Interval begins by defining a **Session Goal**.

Rules:

- The Session Goal must be achievable within one Interval.
- Execution must not begin without a defined Session Goal.
- The Session Goal constrains scope for the Interval.

---

## 5. Bridge

Each Interval must end with a **Bridge**.

A Bridge is a structured continuity artifact that captures execution state for re-entry.

A Bridge must record:

- what was completed
- what remains incomplete
- the next concrete step

Execution may not end without a Bridge.

---

## 6. Modes of Operation

Continuum operates in explicit modes.  
Only one mode may be active at any time.

### 6.1 Decision Mode

Used to select the next story for execution.

### 6.2 Execution Mode

Used to perform work defined by the Session Goal.

Planning and restructuring are prohibited in this mode.

### 6.3 Bridge Mode

Used to produce the Bridge at the end of an Interval.

### 6.4 Alignment Mode

Used to reassess structure, priorities, and assumptions.

---

## 7. Planning Checkpoints

Planning and restructuring are restricted to **Planning Checkpoints**.

A Planning Checkpoint:

- is explicitly scheduled
- allows creation, modification, and reprioritization of stories
- allows restructuring of epics and documentation
- isolates planning activity from execution Intervals

Planning activity is prohibited outside Planning Checkpoints.

---

## 8. Execution Rules

- Only one story may be In Progress at any time.
- Execution must stop when the Session Goal is achieved.
- Missed Intervals do not require compensation.
- Execution resumes by reading the most recent Bridge.

---

## 9. Tooling Role

Tools are used to:

- store execution state
- preserve continuity
- reduce cognitive load

Tools must not:

- imply urgency
- penalize inactivity
- encode shame or performance judgment

---

## 10. Definition of Success

Continuum is functioning correctly if:

- execution can resume without hesitation
- execution state is externally visible
- progress is incremental and consistent
- planning does not interrupt execution
- stopping does not increase future effort

---

## 11. Non-Goals

Continuum does not attempt to:

- optimize velocity
- forecast timelines
- enforce deadlines
- measure productivity
- replace judgment with automation

---

## 12. Invariant

> **Execution must always be easier to resume than to abandon.**

_End of Manifesto_
